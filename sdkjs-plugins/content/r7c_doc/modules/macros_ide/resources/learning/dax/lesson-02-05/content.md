# Урок 8. Логика и обработка пропусков

Вступление

Представьте автопилот самолета, который должен принимать решения в тысячах ситуаций: что делать при отказе датчика высоты, как реагировать на отсутствие сигнала GPS, какое значение использовать при сбое приборов. Без проверок условий и обработки исключений автопилот превратится в источник катастроф. Точно так же в бизнес-аналитике: деление на ноль может исказить все показатели, отсутствующие данные — привести к неверным выводам, а неучтенные условия — к принятию катастрофических решений.

Логические функции и обработка пропусков — это система безопасности ваших DAX-формул. IF и SWITCH позволяют создавать интеллектуальные метрики, адаптирующиеся к данным. DIVIDE защищает от ошибок деления. ISBLANK выявляет отсутствующие значения. SELECTEDVALUE и COALESCE обеспечивают корректные значения по умолчанию. Без этих функций даже простые расчеты могут давать непредсказуемые результаты.

В предыдущих уроках мы научились агрегировать данные и использовать итераторы. Теперь добавим интеллект и надежность нашим формулам через условную логику и грамотную обработку особых случаев.

Теоретическая часть

Основные понятия

Логическое выражение в DAX — это формула, возвращающая TRUE или FALSE. Логические выражения используются для принятия решений в формулах: выполнить одно действие при истинном условии, другое — при ложном. Примеры логических выражений: Sales[Amount] &gt; 1000 (проверка превышения порога), ISBLANK(Customer[Email]) (проверка отсутствия email), Product[Category] = “Bikes” (проверка категории). Результат логического выражения определяет путь выполнения в условных функциях.

Безопасные вычисления — это паттерны DAX, предотвращающие ошибки и некорректные результаты при особых условиях. Классический пример — деление на ноль, которое в обычной математике дает ошибку, но в отчетах должно обрабатываться корректно. Безопасные вычисления включают проверку условий перед операцией, предоставление значений по умолчанию при отсутствии данных, валидацию входных параметров.

Значение по умолчанию — это резервное значение, используемое когда основное вычисление невозможно или возвращает пустой результат. В DAX это может быть ноль вместо BLANK(), текст “Н/Д” вместо ошибки, предыдущее значение при отсутствии текущего. Правильный выбор значений по умолчанию критичен для корректной интерпретации отчетов пользователями.

Каскадная логика — это последовательная проверка множества условий, где каждое следующее условие проверяется только если предыдущие не выполнены. В отличие от вложенных IF, которые создают трудночитаемый код, функция SWITCH(TRUE) позволяет организовать каскадную логику элегантно и понятно.

Ключевая механика

Механизм работы условной логики в DAX основан на коротком замыкании (short-circuit evaluation). Это означает, что DAX прекращает вычисление логического выражения, как только результат становится очевидным. В выражении IF(condition, true_result, false_result), если condition истинно, false_result не вычисляется вообще. Это повышает производительность и позволяет избегать ошибок в невыполняемых ветках.

Представьте условную логику как систему железнодорожных стрелок. Поезд (вычисление) движется по рельсам и на каждой развилке проверяется условие. Стрелка направляет поезд по одному из путей, а альтернативный путь остается неиспользованным. SWITCH(TRUE) — это целая сортировочная станция, где поезд последовательно проверяется на множестве стрелок, пока не найдет подходящий путь.

Обработка BLANK в логических функциях имеет особенности, которые необходимо учитывать. BLANK в логическом контексте преобразуется в FALSE. При сравнениях BLANK = 0 возвращает TRUE для чисел, BLANK = “” возвращает TRUE для текста. Функция ISBLANK — единственный надежный способ явно проверить на BLANK, не полагаясь на автоматические преобразования.

Приоритет выполнения в сложных логических выражениях следует математическим правилам. Сначала выполняются операции в скобках, затем NOT, потом AND (&amp;&amp;), и в последнюю очередь OR (||). Понимание приоритета критично для правильной работы сложных условий. При сомнениях используйте скобки для явного указания порядка.

Контекст выполнения влияет на логические функции так же, как на агрегаты. Условие в IF проверяется в текущем контексте фильтрации. Это означает, что одна и та же логическая формула может давать разные результаты в разных ячейках отчета. Например, IF(SUM(Sales[Amount]) &gt; 10000, “High”, “Low”) будет оценивать разные суммы для разных категорий товаров.

Синтаксис и правила

```dax
IF имеет простой синтаксис: IF(logical_expression, value_if_true, value_if_false). Все три параметра обязательны. Третий параметр можно опустить только если вас устраивает BLANK() при ложном условии. Вложенные IF возможны, но не рекомендуются при более чем 2-3 уровнях вложенности. Пример: IF(Sales[Amount] > 1000, “Large”, IF(Sales[Amount] > 100, “Medium”, “Small”)).
SWITCH в режиме TRUE позволяет проверять множество условий последовательно: SWITCH(TRUE(), condition1, result1, condition2, result2, …, default_result). Первый параметр — всегда TRUE(). Далее идут пары условие-результат. Последний параметр — результат по умолчанию, если ни одно условие не выполнено. Это элегантная замена множественным вложенным IF.
DIVIDE обеспечивает безопасное деление: DIVIDE(numerator, denominator, alternate_result). Если знаменатель равен нулю или BLANK, возвращается alternate_result (по умолчанию BLANK). Это устраняет необходимость проверки деления на ноль через IF. Пример: DIVIDE(Sales[Profit], Sales[Revenue], 0) вернет 0 при нулевой выручке вместо ошибки.
ISBLANK проверяет, является ли значение BLANK: ISBLANK(value). Возвращает TRUE если значение BLANK, FALSE в противном случае. Это единственный надежный способ проверки на BLANK без побочных эффектов автоматического преобразования типов. Часто используется в паре с IF: IF(ISBLANK(Customer[Email]), “No Email”, Customer[Email]).
SELECTEDVALUE возвращает значение, если в текущем контексте выбран ровно один элемент: SELECTEDVALUE(column, alternate_result). Если выбрано 0 или больше 1 элемента, возвращается alternate_result. Полезно для создания динамических заголовков и меток. COALESCE возвращает первое не-BLANK значение из списка: COALESCE(value1, value2, …). Идеально для цепочки значений по умолчанию.
```

Сравнение с альтернативами

В Excel функция ЕСЛИ (IF) работает похоже, но только с ячейками, не с целыми столбцами. Excel также имеет ЕСЛИОШИБКА (IFERROR) для обработки ошибок. В DAX нет прямого аналога IFERROR, но DIVIDE и другие безопасные функции покрывают большинство сценариев. DAX IF мощнее, так как работает с агрегированными данными и учитывает контекст.

SQL использует CASE WHEN для условной логики, что концептуально похоже на SWITCH(TRUE) в DAX. Однако SQL CASE выполняется при запросе данных, а DAX SWITCH — при вычислении меры в отчете. DAX подход более гибкий для интерактивной аналитики, но SQL эффективнее для предварительной обработки больших объемов.

Python pandas использует методы типа fillna() для замены пропущенных значений и np.where() для условной логики. DAX COALESCE похож на fillna(), а IF — на np.where(). Разница в том, что pandas работает с датафреймами в памяти, а DAX — с моделью данных, оптимизированной для аналитических запросов.

Частые ошибки

Забывание обработки BLANK в условиях — источник неожиданных результатов. Условие Sales[Amount] &gt; 100 вернет FALSE для BLANK, хотя логически отсутствие данных не означает “меньше 100”. Всегда явно обрабатывайте BLANK: IF(ISBLANK(Sales[Amount]), “No Data”, IF(Sales[Amount] &gt; 100, “High”, “Low”)).

Использование вложенных IF вместо SWITCH(TRUE) создает нечитаемый код. Три уровня вложенности — это максимум для понимания. Если условий больше, используйте SWITCH(TRUE) для линейной структуры. Код становится читаемым и легко модифицируемым.

Деление без DIVIDE приводит к ошибкам в отчетах. Выражение Sales[Profit] / Sales[Revenue] выбросит ошибку при нулевой выручке. Всегда используйте DIVIDE с указанием альтернативного результата. Это гарантирует стабильность отчетов.

Неправильный порядок условий в SWITCH(TRUE) может привести к неверной логике. Условия проверяются последовательно, и выполняется первое истинное. Размещайте более специфичные условия перед общими. Например, проверка на VIP-клиента должна идти перед проверкой на обычного клиента.
