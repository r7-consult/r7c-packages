# Урок 20. Полусуммируемые показатели

Вступление

Представьте банковский счет с балансом в 1000 долларов на понедельник, 1200 на вторник и 800 на среду. Если сложить эти числа, получится 3000 — но это не означает, что за три дня у вас было 3000 долларов. Баланс на конец периода — 800, а средний — около 1000. Это классический пример полусуммируемого показателя: его можно суммировать по одним измерениям (счета, филиалы), но нельзя по другим (время). В бизнесе такие метрики встречаются повсеместно: остатки на складе, количество сотрудников, активные подписки.

Полусуммируемые показатели требуют специальной логики обработки в DAX. Простое суммирование даст абсурдные результаты, искажающие реальную картину бизнеса. Функции LASTDATE, LASTNONBLANK и специальные паттерны позволяют корректно работать с балансами, остатками и моментальными снимками. Без понимания этих техник невозможно создать точную финансовую или операционную отчетность.

В предыдущих уроках мы изучили различные типы связей и моделирования. Теперь применим эти знания для решения специфической задачи полусуммируемых метрик.

Теоретическая часть

Основные понятия

Полусуммируемые показатели (Semi-Additive Measures) — это метрики, которые можно корректно суммировать только по определенным измерениям, но не по всем. Типичный пример — складские остатки: можно сложить остатки по всем складам на конкретную дату, но нельзя сложить остатки одного склада за разные даты. Сумма дневных остатков не равна общему остатку за период. Полусуммируемость требует специальной логики агрегации в зависимости от контекста.

LASTDATE возвращает последнюю дату из набора дат в текущем фильтровом контексте. Функция полезна для получения значения на конец периода. LASTDATE(Calendar[Date]) в контексте месяца вернет последний день месяца, в контексте года — последний день года. Это ключевая функция для работы с балансами и остатками на конец периода.

LASTNONBLANK находит последнюю дату, для которой указанное выражение возвращает не-BLANK результат. В отличие от LASTDATE, которая просто берет максимальную дату, LASTNONBLANK проверяет наличие данных. LASTNONBLANK(Calendar[Date], SUM(Inventory[Balance])) вернет последнюю дату с непустым балансом. Это критично когда данные поступают нерегулярно.

Моментальный снимок (Snapshot) — это состояние данных на конкретный момент времени. В контексте полусуммируемых показателей, снимки фиксируют значения метрик на определенные даты: конец дня, месяца, квартала. Правильная работа со снимками обеспечивает корректное отображение исторических состояний без искажений от последующих изменений.

Ключевая механика

Механизм работы полусуммируемых показателей основан на выборе правильной стратегии агрегации в зависимости от измерения. При группировке по времени используется последнее или первое значение периода, при группировке по другим измерениям — обычная сумма. DAX должен динамически определять тип агрегации на основе контекста вычисления.

Представьте полусуммируемые показатели как фотографии вместо видео. Вы не можете сложить несколько фотографий, чтобы получить видео — нужно выбрать одну репрезентативную фотографию (обычно последнюю) для представления периода. Но можно объединить фотографии разных объектов, сделанные в один момент, чтобы получить общую картину.

LASTDATE работает как селектор времени, автоматически адаптирующийся к гранулярности отчета. В годовом отчете она выберет 31 декабря, в квартальном — последний день квартала, в месячном — последний день месяца. Это обеспечивает корректное отображение остатков независимо от уровня детализации.

LASTNONBLANK добавляет интеллект к выбору даты, проверяя наличие данных. Если баланс обновляется нерегулярно, простой LASTDATE может вернуть дату без данных, показав BLANK. LASTNONBLANK найдет последнюю дату с реальными данными, обеспечив корректное отображение последнего известного значения.

Производительность полусуммируемых мер может быть проблемой при больших объемах исторических снимков. Каждое вычисление требует поиска правильной даты и извлечения соответствующего значения. Оптимизация через правильное индексирование дат и минимизацию объема снимков критична для скорости отчетов.

Синтаксис и правила

LASTDATE принимает столбец дат: LASTDATE(dates_column). Возвращает таблицу с одной строкой и одним столбцом, содержащую последнюю дату из текущего контекста. Часто используется внутри CALCULATE для фильтрации по последней дате:

```dax
[Balance End of Period] :=
CALCULATE(
SUM(BalanceTable[Amount]),
LASTDATE(Calendar[Date])
)
```

LASTNONBLANK требует два параметра: LASTNONBLANK(dates_column, expression). Первый — столбец дат для проверки, второй — выражение, которое должно быть не-BLANK:

```dax
[Last Balance] :=
CALCULATE(
SUM(BalanceTable[Amount]),
LASTNONBLANK(
Calendar[Date],
CALCULATE(SUM(BalanceTable[Amount]))
)
)
```

Паттерн для полусуммируемых показателей комбинирует обычную сумму для не-временных измерений с выбором последнего значения для времени:

```dax
[Inventory Balance] :=
IF(
HASONEVALUE(Calendar[Date]),
SUM(Inventory[Balance]),  // одна дата - простая сумма
SUMX(
VALUES(Product[ProductID]),
CALCULATE(
SUM(Inventory[Balance]),
LASTDATE(Calendar[Date])
)
)
)
```

Сравнение с альтернативами

В Excel для работы с остатками обычно используются формулы ИНДЕКС/ПОИСКПОЗ или ВПР для поиска последнего значения. Это требует сортировки данных и сложных формул массива. DAX автоматизирует процесс через специализированные функции, адаптирующиеся к контексту.

SQL подходы включают оконные функции с LAST_VALUE или подзапросы с MAX(date). Но SQL вычисляет результат при запросе, не адаптируясь к изменению фильтров. DAX динамически пересчитывает последнее значение при любом изменении контекста.

В традиционных OLAP-кубах полусуммируемые меры настраиваются на уровне метаданных куба. Это менее гибко, чем DAX, где логика определяется в формуле и может быть легко изменена без перестройки всей модели.

Частые ошибки

Использование простого SUM для остатков — самая распространенная ошибка. Сумма дневных остатков не равна остатку на конец периода. Всегда используйте специальную логику для полусуммируемых показателей, выбирая последнее или среднее значение для временной агрегации.

Игнорирование пропусков в данных при использовании LASTDATE может привести к пустым результатам. Если на последнюю дату периода нет данных, мера вернет BLANK. Используйте LASTNONBLANK для поиска последней даты с реальными данными.

Неправильная гранулярность снимков создает искажения. Если снимки делаются нерегулярно или с разной частотой для разных показателей, простые паттерны могут дать некорректные результаты. Убедитесь в консистентности данных снимков.

Смешивание полусуммируемых и обычных показателей в одной формуле приводит к ошибкам. Разделяйте логику для разных типов метрик, создавая отдельные меры с соответствующей логикой агрегации.
