Теоретическая часть

Основные понятия

Пересечение фильтров (Filter Intersection) — это механизм комбинирования множественных условий фильтрации через логическую операцию AND. Когда к данным применяется несколько фильтров одновременно, результирующий набор содержит только те строки, которые удовлетворяют всем условиям. Например, фильтр “Year = 2014” и фильтр “Category = Bikes” создают пересечение, показывающее только велосипеды, проданные в 2014 году. Каждый дополнительный фильтр сужает видимый набор данных, никогда не расширяя его.

Вложенные фильтры (Nested Filters) возникают, когда фильтры применяются на разных уровнях отчета. Срез создает внешний фильтр для всего отчета. Строка сводной таблицы добавляет свой фильтр. Столбец добавляет третий уровень. Эти фильтры не заменяют, а дополняют друг друга, создавая многоуровневую систему фильтрации. Понимание иерархии фильтров критично для предсказания результатов.

Контекст ячейки (Cell Context) — это уникальная комбинация всех фильтров, применяемых к конкретной ячейке отчета. Каждая ячейка сводной таблицы имеет свой контекст, определяемый положением ячейки и активными фильтрами. Контекст ячейки включает фильтры от строки, столбца, срезов и любых других источников фильтрации. Именно контекст ячейки определяет финальное значение меры.

Распространение фильтров (Filter Propagation) — это процесс передачи фильтров через связи модели данных. Когда фильтр применяется к таблице измерения, он автоматически распространяется на связанную фактовую таблицу. Фильтр “Product[Color] = ‘Red’” на таблице продуктов автоматически отфильтрует связанные продажи, оставив только транзакции с красными товарами.

Ключевая механика

Механизм работы фильтрового контекста можно представить как многослойную систему сит. Каждый фильтр — это сито с определенным размером ячеек, пропускающее только подходящие данные. Когда используется несколько фильтров, данные проходят через все сита последовательно. То, что остается после прохождения всех фильтров, становится входными данными для вычисления меры.

Представьте фильтровый контекст как систему ворот в аэропорту. Первые ворота — это срезы отчета (проверка билета). Вторые ворота — фильтр от строки сводной таблицы (паспортный контроль). Третьи — фильтр от столбца (досмотр багажа). Только пассажиры (данные), прошедшие все ворота, попадают в самолет (участвуют в вычислении). Каждые ворота независимы, но все обязательны.

Приоритет фильтров определяется не их источником, а логикой пересечения. Не важно, откуда пришел фильтр — из среза, строки или столбца. Все фильтры равноправны и применяются через AND. Однако, если два фильтра применяются к одному столбцу с разными значениями (например, Color = ‘Red’ AND Color = ‘Blue’), результат будет пустым, так как один товар не может быть одновременно двух цветов.

Связи играют ключевую роль в распространении фильтров. Фильтр применяется к таблице напрямую, затем распространяется на связанные таблицы в направлении связи. По умолчанию фильтры распространяются от стороны “один” к стороне “многие”. Это означает, что фильтр на таблице Customer автоматически отфильтрует связанные Sales, но не наоборот.

Контекстная изоляция означает, что каждая ячейка вычисляется независимо. Значение в одной ячейке не влияет на значение в соседней. Каждая ячейка имеет свой уникальный набор фильтров и вычисляется изолированно. Это обеспечивает предсказуемость и параллелизм вычислений.

Синтаксис и правила

В текущем фильтровом контексте меры автоматически учитывают все активные фильтры без явного указания в формуле. Мера SUM(Sales[Amount]) автоматически суммирует только те строки, которые соответствуют текущему контексту. Нет необходимости писать условия фильтрации — они применяются автоматически.

Для анализа текущего контекста можно использовать функции из уже изученных. COUNTROWS(Sales) покажет количество видимых строк в текущем контексте. DISTINCTCOUNT(Product[ProductKey]) покажет количество уникальных продуктов после применения всех фильтров. Эти функции помогают понять, какие данные доступны мере.

Комбинирование фильтров всегда происходит через логическое AND на уровне разных столбцов. Фильтр Year = 2014 AND Category = ‘Bikes’ оставит только велосипеды 2014 года. Внутри одного среза с множественным выбором используется OR: Category = ‘Bikes’ OR Category = ‘Components’ покажет обе категории.

Пустой результат фильтрации не вызывает ошибку. Если комбинация фильтров не находит подходящих данных, агрегатные функции возвращают BLANK() или 0. SUM возвращает BLANK(), COUNT возвращает 0, AVERAGE возвращает BLANK(). Это поведение можно изменить с помощью IF и ISBLANK.

Сравнение с альтернативами

В Excel фильтры листа работают похоже, но применяются только визуально — скрывают строки, не влияя на формулы напрямую. Функции SUMIF и COUNTIF требуют явного указания условий в каждой формуле. В DAX фильтрация происходит автоматически через контекст, делая формулы универсальными.

SQL WHERE клауза создает явный фильтр для запроса. Вы должны переписывать запрос для каждой комбинации фильтров. DAX меры адаптируются автоматически к любой комбинации фильтров без изменения формулы. Это фундаментальное преимущество для интерактивной аналитики.

Pandas DataFrame в Python использует похожий подход с boolean indexing, где можно комбинировать условия через &amp; и |. Но в pandas фильтрация явная — вы пишете df[condition]. В DAX фильтрация неявная через контекст, что упрощает формулы но требует понимания скрытых механизмов.

Частые ошибки

Ожидание расширения данных при добавлении фильтров — концептуальная ошибка. Новички думают, что дополнительный срез может показать больше данных. В реальности каждый новый фильтр только сужает видимый набор. Если продажи с фильтром “2014 год” показывают $10M, добавление фильтра “Категория Bikes” может только уменьшить эту сумму.

Непонимание пустых результатов при противоречивых фильтрах сбивает с толку. Если выбрать в срезе Country = ‘USA’ и City = ‘London’, результат будет пустым, так как Лондон не находится в США. DAX не выдаст ошибку, просто вернет BLANK() или 0. Важно проверять логическую совместимость фильтров.

Игнорирование направления связей при анализе фильтрации приводит к неожиданностям. Фильтр на фактовой таблице не распространяется автоматически на измерения при однонаправленных связях. Это может создать иллюзию, что фильтр не работает, хотя он просто не распространяется в нужном направлении.

Забывание о множественных путях фильтрации в сложных моделях создает неоднозначности. Если между таблицами есть несколько путей связей, фильтр может распространяться неожиданными маршрутами. Это особенно критично в моделях с циклическими зависимостями.
