# Урок 15: Работа с плагином: модель, связи и проверка мер

Введение: управление моделью данных через плагин

После изучения табличных функций и виртуальных таблиц, пришло время освоить профессиональные инструменты для управления моделью данных. При работе с DAX критически важно правильно настроить связи между таблицами и организовать меры для удобного использования. Ручное управление моделью становится сложным при росте количества таблиц и мер — здесь на помощь приходит плагин «Слайдер данные».

В этом уроке мы научимся использовать плагин не только для выполнения DAX-запросов, но и для полноценного управления табличной моделью. Вы узнаете, как импортировать таблицы, настраивать связи между ними, проверять гранулярность данных и организовывать меры в логические группы. Особое внимание уделим диагностике проблем со связями — частой причине неправильных результатов в DAX-формулах.

Основное преимущество работы через плагин — возможность видеть структуру модели визуально, быстро тестировать связи и валидировать результаты мер прямо в Excel. Это экономит время и помогает избежать ошибок при разработке сложных аналитических решений.

Важная особенность: плагин автоматически определяет потенциальные ключи для связей на основе имен столбцов и типов данных. Это упрощает настройку связей, но всегда требует ручной проверки.

## Проверка импортированных данных

## После импорта всегда выполняйте базовую проверку с помощью простых DAX-мер

```dax
// Проверка количества строк
[Row Count] := COUNTROWS(DimProduct)
// Проверка уникальности ключа
[Unique Keys] := DISTINCTCOUNT(DimProduct[ProductKey])
// Проверка на дубликаты (должно быть 0)
[Duplicates] := [Row Count] - [Unique Keys]
```

## Проверка направления фильтрации

Направление фильтрации критично для правильной работы DAX-формул. Создайте тестовые меры для проверки:

```dax
// Тест прямой фильтрации (от DimProduct к FactInternetSales)
[Products with Sales] :=
CALCULATE(
DISTINCTCOUNT(DimProduct[ProductKey]),
FactInternetSales
)
// Тест обратной фильтрации (нужна bi-directional связь)
[Sales Categories] :=
CALCULATE(
DISTINCTCOUNT(FactInternetSales[SalesOrderNumber]),
DimProduct[Category] = "Bikes"
)
```

Диагностика проблем со связями

## Проверка на Many-to-Many

Many-to-Many связи часто приводят к неправильным результатам. Используйте эту меру для диагностики:

```dax
// Проверка уникальности ключа связи
[Check Relationship] :=
VAR KeyCount = DISTINCTCOUNT(DimProduct[ProductKey])
VAR RowCount = COUNTROWS(DimProduct)
RETURN
IF(KeyCount < RowCount, "Duplicate Keys Found!", "OK")
```

## Проверка неактивных связей

Иногда связи могут быть неактивными из-за конфликтов. Плагин подсвечивает такие связи серым цветом. Для их активации используйте USERELATIONSHIP в DAX.

## Пример тестовой конфигурации

```dax
// Базовая мера для тестирования
[Total Sales] := SUM(FactInternetSales[SalesAmount])
// Проверка с фильтром по категории
[Bikes Sales] :=
CALCULATE(
[Total Sales],
DimProduct[Category] = "Bikes"
)
// Проверка временной intelligence
[YTD Sales] :=
CALCULATE(
[Total Sales],
DATESYTD(DimDate[Date])
)
```

Проверка контекста фильтрации

## Создайте специальные меры для диагностики контекста

```dax
// Показывает текущий контекст фильтра
[Current Context] :=
CONCATENATEX(
VALUES(DimProduct[Category]),
DimProduct[Category],
", "
)
// Количество строк в текущем контексте
[Context Rows] := COUNTROWS(FactInternetSales)
```

Валидация результатов

## Всегда создавайте проверочные меры для валидации

```dax
// Проверка суммы по частям
[Check Total] :=
VAR DetailSum = SUMX(VALUES(DimProduct[Category]), [Total Sales])
VAR GrandTotal = CALCULATE([Total Sales], ALL(DimProduct[Category]))
RETURN
IF(DetailSum = GrandTotal, "Valid", "Error: " & (GrandTotal - DetailSum))
```

Упражнение 2: Проверка гранулярности через меры

Задача: Определите гранулярность таблицы FactInternetSales и проверьте связи.

## Создайте набор диагностических мер

```dax
// Гранулярность по дате
[Orders per Day] :=
AVERAGEX(
VALUES(DimDate[Date]),
CALCULATE(COUNTROWS(FactInternetSales))
)
// Гранулярность по продукту
[Sales per Product] :=
AVERAGEX(
VALUES(DimProduct[ProductKey]),
CALCULATE(SUM(FactInternetSales[SalesAmount]))
)
// Проверка полноты данных
[Days with Sales] :=
CALCULATE(
DISTINCTCOUNT(DimDate[Date]),
FactInternetSales
)
```

Упражнение 3: Создание шаблона отчета

Задача: Создайте переиспользуемый шаблон отчета с базовыми метриками.

## Структура шаблона

```dax
// === БАЗОВЫЕ МЕРЫ ===
[Total Amount] := SUM(FactInternetSales[SalesAmount])
[Total Quantity] := SUM(FactInternetSales[OrderQuantity])
[Order Count] := DISTINCTCOUNT(FactInternetSales[SalesOrderNumber])
// === СРЕДНИЕ ПОКАЗАТЕЛИ ===
[Avg Order Value] := DIVIDE([Total Amount], [Order Count])
[Avg Items per Order] := DIVIDE([Total Quantity], [Order Count])
// === ВРЕМЕННЫЕ СРАВНЕНИЯ ===
[MTD Amount] :=
CALCULATE(
[Total Amount],
DATESMTD(DimDate[Date])
)
[Prior Month Amount] :=
CALCULATE(
[Total Amount],
DATEADD(DimDate[Date], -1, MONTH)
)
[MoM Growth %] :=
DIVIDE(
[Total Amount] - [Prior Month Amount],
[Prior Month Amount]
)
```

Сохраните этот набор мер как шаблон через «Export Measures» для использования в других проектах.
