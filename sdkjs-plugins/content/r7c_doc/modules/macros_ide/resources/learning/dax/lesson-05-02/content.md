# Урок 18. Управление связями в формулах

Вступление

Представьте железнодорожную станцию с множеством путей, где поезда могут идти только по активным рельсам, а некоторые пути временно закрыты для ремонта. Диспетчер может переключать стрелки, направляя поезда по альтернативным маршрутам. В DAX связи между таблицами работают похожим образом — есть активные и неактивные связи, и вы можете программно переключаться между ними. USERELATIONSHIP и CROSSFILTER дают полный контроль над тем, как данные соединяются и фильтруются в каждой конкретной формуле.

Управление связями критично для сложных моделей с множественными датами, альтернативными иерархиями или особыми бизнес-правилами. Без этих функций пришлось бы создавать дублирующие таблицы или усложнять модель избыточными связями. Возможность активировать нужную связь именно в момент вычисления открывает путь к элегантным решениям сложнейших аналитических задач.

В предыдущем уроке мы изучили принципы качественной модели данных. Теперь научимся управлять связями динамически для создания гибких аналитических сценариев.

Теоретическая часть

Основные понятия

```dax
USERELATIONSHIP — функция активации неактивной связи между таблицами для конкретного вычисления. В модели может быть только одна активная связь между двумя таблицами, но можно создать дополнительные неактивные связи. USERELATIONSHIP временно активирует указанную связь внутри CALCULATE, позволяя использовать альтернативные пути соединения. Например, CALCULATE(SUM(Sales[Amount]), USERELATIONSHIP(Sales[ShipDate], Calendar[Date])) использует дату отгрузки вместо даты заказа.
```

CROSSFILTER изменяет направление фильтрации существующей связи или временно отключает её. Функция принимает два столбца связи и направление: NONE (отключить), ONEWAY (однонаправленная), BOTH (двунаправленная). CROSSFILTER работает только внутри CALCULATE и влияет на все вычисления в этом контексте. Это позволяет динамически изменять поведение модели без изменения физических связей.

Ролевые даты (Role-Playing Dates) — это концепция использования одной таблицы календаря для разных дат в фактовой таблице. Заказ может иметь дату создания, дату оплаты, дату отгрузки. Вместо трех календарей используется один с тремя связями, из которых активна только одна. USERELATIONSHIP позволяет переключаться между этими датами в зависимости от аналитической задачи.

Направление фильтрации определяет, как фильтры распространяются через связь. При однонаправленной фильтрации (по умолчанию) фильтр идет от таблицы “один” к таблице “многие”. Двунаправленная фильтрация позволяет фильтрам идти в обоих направлениях, что полезно для некоторых сценариев, но может создавать неоднозначности и влиять на производительность.

Ключевая механика

Механизм работы USERELATIONSHIP основан на временной замене активной связи. При вычислении выражения внутри CALCULATE с USERELATIONSHIP система деактивирует текущую активную связь между указанными таблицами и активирует альтернативную. Это влияет на все вычисления внутри данного CALCULATE, включая вложенные функции. После завершения вычисления исходная связь восстанавливается.

Представьте USERELATIONSHIP как временный мост через реку. Обычно все переходят по главному мосту (активная связь), но иногда нужно использовать запасной мост выше по течению (неактивная связь). USERELATIONSHIP временно закрывает главный мост и открывает альтернативный, направляя весь трафик по новому пути. После вычисления главный мост снова открывается.

CROSSFILTER работает как регулятор направления движения на дороге. По умолчанию движение однонаправленное — от измерений к фактам. CROSSFILTER может сделать дорогу двусторонней (BOTH), позволяя фильтрам идти обратно от фактов к измерениям, или полностью перекрыть дорогу (NONE), изолировав таблицы друг от друга.

Взаимодействие множественных связей требует понимания приоритетов. Если между таблицами есть несколько путей фильтрации через промежуточные таблицы, система выбирает кратчайший активный путь. USERELATIONSHIP может изменить этот путь, активировав альтернативную прямую связь. Важно помнить: в каждый момент времени между двумя таблицами может быть активна только одна прямая связь.

Влияние на производительность зависит от типа изменений. Активация неактивной связи через USERELATIONSHIP обычно не влияет на скорость. Но включение двунаправленной фильтрации через CROSSFILTER может существенно замедлить вычисления, особенно при высокой кардинальности. Система должна проверять фильтры в обоих направлениях, что увеличивает вычислительную нагрузку.

Синтаксис и правила

```dax
USERELATIONSHIP имеет простой синтаксис: USERELATIONSHIP(column1, column2). Оба параметра — столбцы, участвующие в неактивной связи. Функция работает только внутри CALCULATE или CALCULATETABLE. Пример:
[Sales by Ship Date] :=
CALCULATE(
SUM(FactInternetSales[SalesAmount]),
USERELATIONSHIP(FactInternetSales[ShipDateKey], DimDate[DateKey])
)
```

CROSSFILTER требует три параметра: CROSSFILTER(column1, column2, direction). Первые два — столбцы существующей связи, третий — направление (NONE, ONEWAY, BOTH). Функция также работает только внутри CALCULATE:

```dax
[Bidirectional Filter] :=
CALCULATE(
COUNTROWS(DimProduct),
CROSSFILTER(FactInternetSales[ProductKey], DimProduct[ProductKey], BOTH)
)
```

Важное правило: USERELATIONSHIP работает только с существующими неактивными связями. Нельзя создать связь “на лету” — она должна быть предварительно определена в модели. Если связи нет, возникнет ошибка.

При использовании нескольких USERELATIONSHIP в одном CALCULATE каждая функция влияет на свою пару таблиц независимо. Можно одновременно переключить несколько связей:

```dax
CALCULATE(
[Measure],
USERELATIONSHIP(Sales[ShipDate], Calendar[Date]),
USERELATIONSHIP(Sales[CustomerID], AlternativeCustomers[ID])
)
```

Сравнение с альтернативами

В традиционных базах данных для работы с множественными связями используются алиасы таблиц и явные JOIN в каждом запросе. DAX-подход с предопределенными связями и их активацией более декларативен и понятен для бизнес-пользователей. Не нужно каждый раз писать условия соединения.

Excel не имеет концепции множественных связей между таблицами. Для анализа по разным датам пришлось бы дублировать данные или создавать сложные формулы с множественными условиями. DAX решает это элегантно через переключение связей.

Power Query может создавать соединения при загрузке, но они статичны. Изменение типа соединения требует перезагрузки данных. USERELATIONSHIP и CROSSFILTER работают динамически, позволяя менять логику соединения в реальном времени без изменения данных.

Частые ошибки

Попытка использовать USERELATIONSHIP без CALCULATE — синтаксическая ошибка. Функция работает только как модификатор фильтра внутри CALCULATE. Вне CALCULATE она не имеет эффекта и вызовет ошибку компиляции.

Забывание о деактивации основной связи при использовании USERELATIONSHIP приводит к неожиданным результатам. Когда активируется альтернативная связь, основная автоматически деактивируется. Это может нарушить другие вычисления, опирающиеся на основную связь.

Избыточное использование двунаправленной фильтрации замедляет модель. Не делайте все связи двунаправленными “на всякий случай”. Используйте CROSSFILTER точечно, только где это действительно необходимо для конкретного вычисления.

Создание циклических зависимостей при изменении направления фильтрации может привести к неоднозначностям. Если включить двунаправленную фильтрацию на всех связях в цикле, система не сможет определить приоритет фильтрации.

Ответы

Обе функции работают только внутри CALCULATE или CALCULATETABLE как модификаторы контекста.

Ролевые даты — использование одного календаря для разных дат через множественные связи с переключением.

Нет, между двумя таблицами может быть активна только одна прямая связь в каждый момент.
