Теоретическая часть

Основные понятия

Звездообразная схема (Star Schema) — это структура модели данных, где центральная таблица фактов окружена таблицами измерений, создавая визуальную форму звезды. Таблица фактов содержит числовые метрики и ключи для связи с измерениями. Измерения содержат описательные атрибуты для анализа. Например, FactInternetSales (факты) связана с DimProduct, DimCustomer, DimDate (измерения). Каждое измерение связано с фактами напрямую, без промежуточных таблиц, что упрощает навигацию и ускоряет запросы.

Денормализация измерений — это процесс объединения связанных таблиц в одну для упрощения модели. Вместо отдельных таблиц Product, Subcategory, Category создается одна таблица Product со всеми атрибутами. Это увеличивает размер измерения из-за дублирования данных, но устраняет необходимость в дополнительных связях, упрощает формулы DAX и улучшает производительность запросов.

Кардинальность (Cardinality) — это количество уникальных значений в столбце. Низкая кардинальность означает мало уникальных значений (пол: М/Ж), высокая — много (номер паспорта). Кардинальность критически влияет на производительность: столбцы с высокой кардинальностью требуют больше памяти для хранения и времени для обработки. Оптимальная модель минимизирует высококардинальные связи.

Направление фильтрации определяет, как фильтры распространяются между таблицами. По умолчанию фильтры идут от стороны “один” к стороне “многие” (однонаправленная фильтрация). Двунаправленная фильтрация позволяет фильтрам распространяться в обоих направлениях, но может создавать неоднозначности и замедлять модель. Правильный выбор направления критичен для корректности вычислений.

Ключевая механика

Механизм работы звездообразной схемы основан на принципе единой точки истины для каждого атрибута. Каждый факт хранится только в таблице фактов, каждый атрибут измерения — только в соответствующей таблице измерения. Это исключает противоречия и упрощает обновление данных. Связи всегда идут от измерений (один) к фактам (многие), создавая естественную иерархию фильтрации.

Представьте звездообразную схему как солнечную систему. Солнце (таблица фактов) находится в центре, планеты (измерения) вращаются вокруг него. Каждая планета независима от других, но все связаны с солнцем. Фильтр на планете (например, выбор категории продукта) влияет на солнце (фильтрует факты), но не влияет напрямую на другие планеты. Это обеспечивает предсказуемость и независимость измерений.

Денормализация работает как предварительная компиляция кода. Вместо того чтобы каждый раз собирать информацию из нескольких таблиц через связи, все атрибуты уже собраны в одном месте. Это ускоряет доступ к данным, но требует больше места для хранения. В контексте колоночного хранения VertiPaq дублирование текстов эффективно сжимается, минимизируя недостатки денормализации.

Влияние кардинальности на производительность нелинейно. Столбец с 10 уникальными значениями обрабатывается почти мгновенно. Столбец с 10,000 значениями работает приемлемо. Но столбец с 1,000,000 уникальных значений может замедлить модель в десятки раз. Каждое уникальное значение требует отдельной записи в словаре сжатия, увеличивая объем памяти и время поиска.

Уникальные ключи обеспечивают целостность связей. Сторона “один” в связи должна содержать только уникальные значения. Нарушение уникальности приводит к дублированию данных при соединении и некорректным результатам агрегации. Power Pivot автоматически проверяет уникальность при создании связи, но важно поддерживать это свойство при обновлении данных.

Синтаксис и правила

Проверка кардинальности выполняется через простые меры DAX. Для анализа уникальности используйте DISTINCTCOUNT:

```dax
[Product Key Cardinality] := DISTINCTCOUNT(DimProduct[ProductKey])
[Check Uniqueness] := IF(COUNTROWS(DimProduct) = DISTINCTCOUNT(DimProduct[ProductKey]), "Unique", "Duplicates")
```

## Денормализация в DAX может быть проверена через анализ связанных атрибутов

```dax
[Category From Product] := VALUES(DimProduct[Category])
```

Если измерение правильно денормализовано, все атрибуты доступны из одной таблицы без дополнительных связей.

## Направление фильтрации можно проверить через тестовые меры

```dax
[Filter Direction Test] := COUNTROWS(DimProduct)  // в контексте FactInternetSales
```

Если мера возвращает отфильтрованное количество продуктов при выборе клиента, значит есть путь фильтрации через факты.

Сравнение с альтернативами

В традиционных реляционных базах данных используется нормализация для устранения избыточности. Это оптимально для транзакционных систем, где важна целостность при обновлении. Но для аналитики денормализованная звезда эффективнее — меньше соединений, проще запросы, быстрее результаты.

Снежинка (Snowflake Schema) — альтернатива звезде с нормализованными измерениями. Product связан с Subcategory, которая связана с Category. Это экономит место, но усложняет модель и замедляет запросы. В контексте DAX и колоночного сжатия преимущества снежинки минимальны, а недостатки существенны.

Плоская таблица (одна большая таблица со всеми данными) кажется простым решением, но создает огромные проблемы. Массовое дублирование данных, невозможность независимого обновления измерений, сложность в управлении. Звезда — оптимальный баланс между простотой и эффективностью.

Частые ошибки

Создание связей с высококардинальными столбцами (например, по номеру транзакции) катастрофически замедляет модель. Каждое уникальное значение требует отдельной обработки при фильтрации. Используйте агрегированные ключи или избегайте таких связей, выполняя соединение через виртуальные таблицы.

Циклические зависимости в связях создают неоднозначности. Если Product связан с Sales, Sales с Customer, Customer с Product через предпочтения, образуется цикл. Power Pivot не позволит активировать все связи одновременно. Планируйте модель как дерево, а не граф.

Отсутствие проверки уникальности ключей после обновления данных приводит к скрытым ошибкам. Новые данные могут нарушить уникальность, создавая дубликаты. Всегда включайте проверочные меры для контроля качества модели после обновления.

Чрезмерная денормализация с включением всех возможных атрибутов раздувает измерения. Не каждый атрибут нужен в модели. Включайте только те, которые используются для анализа. Технические поля и метаданные лучше исключить.

Ответы

Звездообразная схема — структура с центральной таблицей фактов и окружающими измерениями, оптимальна для быстрой фильтрации и простоты.

Денормализация устраняет дополнительные связи, упрощает формулы и ускоряет запросы за счет предварительного объединения атрибутов.

Высокая кардинальность требует больше памяти и времени обработки, каждое уникальное значение увеличивает нагрузку на движок.

Уникальность ключей предотвращает дублирование при соединении и обеспечивает корректность агрегации данных.

Фильтры должны идти от измерений (один) к фактам (многие), обеспечивая предсказуемость и изоляцию измерений.

Циклические зависимости создают неоднозначность путей фильтрации и могут привести к непредсказуемым результатам.

Звезда имеет денормализованные измерения (проще), снежинка — нормализованные (больше таблиц, сложнее запросы).
