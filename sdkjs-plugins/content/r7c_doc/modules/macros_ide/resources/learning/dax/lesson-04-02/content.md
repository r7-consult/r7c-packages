# Урок 14. Календарь: фундамент для Time Intelligence

Вступление

Представьте финансового директора, которому нужно сравнить продажи текущего квартала с прошлым годом, рассчитать накопительный итог с начала года и спрогнозировать сезонные колебания. Без правильно настроенной календарной таблицы эти задачи превращаются в кошмар из сотен формул. Календарь в DAX — это не просто список дат, а интеллектуальная структура, понимающая иерархию времени, рабочие и выходные дни, финансовые периоды. Это фундамент, на котором строится вся временная аналитика.

Календарная таблица критически важна для любой бизнес-модели. Без нее невозможны корректные сравнения периодов, расчет трендов, анализ сезонности. DAX предоставляет специальные функции для создания календарей, но правильная настройка требует понимания бизнес-требований и технических нюансов. От структуры календаря зависит корректность всех временных вычислений.

Теоретическая часть

Основные понятия

CALENDAR — функция создания календарной таблицы с указанным диапазоном дат. CALENDAR(start_date, end_date) генерирует таблицу с одним столбцом [Date], содержащим непрерывную последовательность дат от начальной до конечной включительно. Например, CALENDAR(DATE(2020,1,1), DATE(2023,12,31)) создаст таблицу с каждым днем за четыре года. Это основа для построения полноценного календарного измерения с дополнительными атрибутами.

CALENDARAUTO — интеллектуальная функция автоматического создания календаря на основе дат в модели данных. CALENDARAUTO() сканирует все столбцы дат во всех таблицах модели, находит минимальную и максимальную даты, создает календарь, покрывающий полные годы. Опциональный параметр fiscal_year_end_month позволяет указать последний месяц финансового года. CALENDARAUTO(6) создаст календарь с финансовым годом, заканчивающимся в июне.

Порядковые ключи — числовые атрибуты для правильной сортировки временных элементов. Месяцы должны сортироваться не алфавитно (April, August…), а хронологически (January, February…). Для этого нужен столбец MonthNumber (1-12). Аналогично для кварталов (Q1-Q4) нужен QuarterNumber (1-4). Порядковые ключи критичны для корректной визуализации временных данных.

Mark as Date Table — специальная настройка в Power Pivot, указывающая системе, что таблица является календарным измерением. Это активирует автоматическое распознавание временных иерархий и обеспечивает корректную работу функций Time Intelligence. Без этой настройки многие временные функции DAX работают некорректно или возвращают ошибки.

Ключевая механика

Механизм работы календарной таблицы основуума без пропусков. Даже если в данных нет продаж за определенные даты, календарь должен содержать эти даты. Это обеспечивает корректные вычисления при сравнении периодов и расчете трендов. Пропуск даты в календаре может привести к искажению накопительных итогов и неверным процентным изменениям.

Представьте календарь как координатную ось времени в вашей модели данных. Каждая точка на оси — это день с полным набором атрибутов: к какому месяцу относится, какой это день недели, рабочий или выходной, к какому кварталу и году принадлежит. Факты (продажи, заказы) привязываются к этой оси через даты, создавая временную картину бизнеса.

Иерархическая структура календаря критична для анализа на разных уровнях детализации. Год содержит кварталы, кварталы содержат месяцы, месяцы содержат дни. Эта иерархия должна быть явно определена через соответствующие столбцы. DAX автоматически агрегирует данные вверх по иерархии: сумма дней дает месяц, сумма месяцев — квартал, сумма кварталов — год.

Связь календаря с фактами должна быть однозначной и основываться на столбцах типа Date. Распространенная ошибка — связывание по текстовым представлениям дат или составным ключам. Это нарушает работу Time Intelligence функций. Правильная связь: Calendar[Date] → FactTable[OrderDate], где оба столбца имеют тип Date.

Обогащение календаря бизнес-атрибутами делает его универсальным инструментом анализа. Добавление флагов рабочих дней позволяет корректно считать бизнес-метрики. Маркировка праздников объясняет аномалии в данных. Финансовые периоды обеспечивают корректную отчетность. Каждый атрибут расширяет аналитические возможности.

Синтаксис и правила

Базовый синтаксис CALENDAR: CALENDAR(start_date, end_date). Оба параметра обязательны и должны быть типа Date. Можно использовать функцию DATE для явного указания: CALENDAR(DATE(2020,1,1), DATE(2023,12,31)). Результат — таблица с одним столбцом [Date]. Пример полного календаря за 2020-2023:

```dax
CalendarTable = CALENDAR(DATE(2020,1,1), DATE(2023,12,31))
```

CALENDARAUTO имеет опциональный параметр: CALENDARAUTO([fiscal_year_end_month]). Без параметра создается календарный год (январь-декабрь). С параметром — финансовый год, заканчивающийся указанным месяцем. CALENDARAUTO(3) создаст финансовый год апрель-март. Функция автоматически определяет необходимый диапазон дат.

Обогащение календаря выполняется через ADDCOLUMNS после создания базовой таблицы. Типовая структура:

```dax
Calendar =
ADDCOLUMNS(
CALENDAR(...),
"Year", YEAR([Date]),
"Month", FORMAT([Date], "MMMM"),
"MonthNumber", MONTH([Date])
)
```

Важное правило: календарь должен содержать все обязательные столбцы для Time Intelligence: Date (уникальная дата), Year, Month, MonthNumber, Quarter, QuarterNumber. Без этих столбцов многие временные функции не работают. Дополнительные столбцы опциональны, но расширяют возможности анализа.

Сравнение с альтернативами

В Excel календарь обычно создается вручную или через автозаполнение. Это трудоемко и подвержено ошибкам при обновлении диапазона дат. DAX автоматизирует процесс, гарантируя полноту и непрерывность календаря. Изменение диапазона требует только корректировки параметров функции.

SQL-подход требует создания отдельной таблицы дат в базе данных с заполнением через скрипты. Это эффективно для статичных периодов, но негибко при изменении требований. DAX создает календарь динамически при загрузке модели, автоматически адаптируясь к данным.

Power Query может генерировать календари через М-код. Это дает больше контроля над процессом создания, но требует знания другого языка. DAX-подход проще для понимания и интеграции с остальными вычислениями модели.

Частые ошибки

Создание календаря с пропусками дат нарушает работу Time Intelligence. Если календарь содержит только даты с транзакциями, функции сравнения периодов дают некорректные результаты. Всегда создавайте непрерывный календарь, даже если часть дат не имеет данных.

Отсутствие порядковых ключей приводит к неправильной сортировке в визуализациях. Месяцы сортируются алфавитно, кварталы — по тексту. Всегда добавляйте числовые ключи для хронологической сортировки. Настройте сортировку текстовых столбцов по соответствующим числовым.

Множественные календари в модели создают путаницу и ошибки. Используйте один мастер-календарь для всех фактовых таблиц. При необходимости разных календарей (например, для заказов и поставок) четко именуйте их и документируйте назначение.

Неправильный тип данных в столбце дат нарушает связи и Time Intelligence. Убедитесь, что столбец Date имеет тип Date, а не Text или Number. Проверьте типы данных в связанных таблицах — они должны совпадать.

## DAX-решение

```dax
// Создание базового календаря (вычисляемая таблица)
MyCalendar =
ADDCOLUMNS(
CALENDAR(DATE(2010,1,1), DATE(2014,12,31)),  // 5 полных лет
"Year", YEAR([Date]),  // год
"MonthName", FORMAT([Date], "MMMM"),  // название месяца
"MonthNumber", MONTH([Date]),  // номер месяца для сортировки
"Quarter", "Q" & QUARTER([Date]),  // квартал
"QuarterNumber", QUARTER([Date])  // номер квартала
)
// Проверка диапазона календаря
[Calendar Days Count] :=
COUNTROWS(MyCalendar)  // должно быть 1826 дней (5 лет)
```

Проверка результата: Календарь содержит 1,826 строк (365 × 5 + 1 високосный день) Годы: 2010-2014, по 12 месяцев в каждом Кварталы: Q1-Q4 для каждого года При связывании с FactInternetSales все транзакции находят соответствие Проверка суммы продаж по годам: 2010 - $0, 2011 - $12,450,596.98, 2012 - $6,865,064.90

## DAX-решение

```dax
// Автоматический финансовый календарь
FiscalCalendar =
ADDCOLUMNS(
CALENDARAUTO(6),  // финансовый год заканчивается в июне
"FiscalYear", IF(MONTH([Date]) > 6, YEAR([Date]) + 1, YEAR([Date])),
"FiscalQuarter", "FQ" & CEILING(MONTH([Date]) - 6, 3) / 3,
"IsCurrentFY", IF(YEAR([Date]) = 2014 && MONTH([Date]) <= 6, "Current", "Past")
)
// Проверка автоматического диапазона
[Auto Calendar Range] :=
"From " & MIN(FiscalCalendar[Date]) & " to " & MAX(FiscalCalendar[Date])
```

Проверка результата: CALENDARAUTO создал календарь с 01.07.2010 по 30.06.2015 (полные финансовые годы) Финансовый 2011: июль 2010 - июнь 2011 Финансовый 2015: июль 2014 - июнь 2015 Всего строк: 1,826 (полные годы) Продажи по финансовым годам корректно агрегируются

## DAX-решение

```dax
// Расширенный бизнес-календарь
BusinessCalendar =
ADDCOLUMNS(
CALENDAR(DATE(2010,1,1), DATE(2014,12,31)),
"Year", YEAR([Date]),
"Month", FORMAT([Date], "MMMM"),
"MonthNumber", MONTH([Date]),
"DayOfWeek", FORMAT([Date], "dddd"),  // название дня недели
"DayOfWeekNum", WEEKDAY([Date], 2),  // 1=пон, 7=вос
"IsWorkday", IF(WEEKDAY([Date], 2) <= 5, 1, 0),  // рабочий день
"WeekNumber", WEEKNUM([Date]),  // номер недели
"YearMonth", FORMAT([Date], "YYYY-MM")  // ключ для группировки
)
// Количество рабочих дней в контексте
[Workdays Count] :=
SUMX(BusinessCalendar, BusinessCalendar[IsWorkday])
```

Проверка результата: Общее количество дней: 1,826 Рабочих дней: 1,304 (71.4%) Выходных: 522 (28.6%) 2013 год: 261 рабочий день, 104 выходных Январь 2013: 23 рабочих дня, 8 выходных Продажи в рабочие дни: $24,567,890.12 (83.7% от общих)

Ответы

Обе функции возвращают таблицу с одним столбцом [Date], содержащим последовательность дат в указанном диапазоне.

Обязательные столбцы: Date (уникальная дата), Year, Month, MonthNumber, Quarter, QuarterNumber для корректной работы временных функций.

Порядковые ключи (MonthNumber, QuarterNumber) обеспечивают правильную хронологическую сортировку вместо алфавитной.

Параметр 6 означает, что финансовый год заканчивается в июне (6-й месяц), следующий начинается с июля.

Непрерывность календаря обеспечивает корректные вычисления при сравнении периодов и расчете накопительных итогов.

“Mark as Date Table” указывает Power Pivot, что таблица является временным измерением, активируя автоматические временные иерархии.

Можно, но это усложняет модель; рекомендуется один мастер-календарь для всех фактовых таблиц.
