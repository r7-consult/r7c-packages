# Урок 16. Виртуальные таблицы и соединения

Вступление

Представьте архитектора данных, которому нужно объединить информацию из пяти разных источников, создать все возможные комбинации товаров и регионов, найти пересечения клиентских баз двух филиалов и исключить дубликаты. В традиционных базах данных это потребовало бы создания десятков промежуточных таблиц. В DAX функции соединения создают виртуальные таблицы прямо в формулах — мощные, гибкие структуры, существующие только во время вычисления. CROSSJOIN, UNION, INTERSECT открывают возможности для сложнейшей аналитики без изменения физической модели.

Виртуальные таблицы и соединения — это вершина табличных вычислений в DAX. Они позволяют создавать динамические наборы данных, комбинировать информацию из несвязанных источников, выполнять теоретико-множественные операции. Это инструменты для решения задач, невозможных с базовыми функциями: анализ корзин покупок, матричные вычисления, сложные сегментации.

В предыдущих уроках мы изучили группировки и календари. Теперь освоим продвинутые операции с таблицами для создания сложных аналитических конструкций.

Теоретическая часть

Основные понятия

```dax
CROSSJOIN создает декартово произведение двух или более таблиц — каждая строка первой таблицы комбинируется с каждой строкой второй. Если первая таблица содержит 10 строк, а вторая 5, результат будет содержать 50 строк со всеми возможными комбинациями. CROSSJOIN(Products, Regions) создаст виртуальную таблицу со всеми теоретически возможными комбинациями продукт-регион, даже если некоторые продукты не продаются в некоторых регионах. Это основа для матричных вычислений и анализа покрытия.
```

UNION объединяет строки из двух или более таблиц с одинаковой структурой столбцов. В отличие от SQL, DAX UNION не удаляет дубликаты автоматически — для этого нужно обернуть результат в DISTINCT. UNION(Sales2019, Sales2020, Sales2021) создаст единую таблицу со всеми продажами за три года. Важно: таблицы должны иметь одинаковое количество столбцов с совместимыми типами данных.

INTERSECT возвращает строки, присутствующие во всех входных таблицах — пересечение множеств. EXCEPT возвращает строки из первой таблицы, отсутствующие во второй — разность множеств. Эти функции полезны для сравнительного анализа: INTERSECT(CustomersUSA, CustomersVIP) найдет VIP-клиентов из США, EXCEPT(AllProducts, SoldProducts) покажет непроданные товары.

Естественные соединения NATURALINNERJOIN и NATURALLEFTOUTERJOIN автоматически объединяют таблицы по общим столбцам. “Естественное” означает, что DAX сам определяет столбцы для соединения по совпадению имен. INNER возвращает только строки с совпадениями в обеих таблицах, LEFTOUTERJOIN сохраняет все строки из левой таблицы, дополняя BLANK для отсутствующих справа.

Ключевая механика

Механизм работы виртуальных таблиц основан на создании временных структур в памяти во время вычисления. Эти таблицы не сохраняются в модели, не занимают постоянную память, существуют только пока выполняется формула. После получения результата виртуальная таблица уничтожается. Это позволяет создавать сложные промежуточные вычисления без раздувания модели.

Представьте виртуальные таблицы как чертежи на песке — вы рисуете сложную схему, используете ее для вычислений, затем волна (завершение формулы) смывает рисунок. Следующее вычисление рисует новую схему. В отличие от физических таблиц, высеченных в камне, виртуальные таблицы гибкие и не требуют места для хранения.

CROSSJOIN работает как вложенные циклы: для каждой строки первой таблицы добавляются все строки второй. Это создает комбинаторный взрыв — количество строк растет мультипликативно. С тремя таблицами по 100 строк CROSSJOIN создаст миллион комбинаций. Поэтому критично фильтровать входные таблицы перед применением CROSSJOIN.

Операции множеств (INTERSECT, EXCEPT, UNION) работают на уровне полных строк, а не отдельных значений. Две строки считаются одинаковыми, только если совпадают все значения во всех столбцах. Это отличается от сравнения по ключу в обычных соединениях. Порядок столбцов важен — таблицы должны иметь идентичную структуру.

Естественные соединения интеллектуально сопоставляют столбцы по именам. Если таблицы имеют столбцы ProductID и CustomerID, соединение произойдет по обоим. Это удобно, но может привести к неожиданным результатам при случайном совпадении имен. Всегда проверяйте структуру таблиц перед естественным соединением.

Синтаксис и правила

```dax
CROSSJOIN принимает две или более таблиц: CROSSJOIN(table1, table2, …). Таблицы могут быть физическими или результатом других функций. Количество столбцов в результате равно сумме столбцов входных таблиц. Пример: CROSSJOIN(VALUES(Product[Color]), VALUES(Product[Size])) создаст все комбинации цветов и размеров.
```

UNION требует таблиц с одинаковым количеством столбцов: UNION(table1, table2, …). Типы данных должны быть совместимы для соответствующих позиций. Имена столбцов берутся из первой таблицы. Пример: UNION(FILTER(Sales, Sales[Year] = 2019), FILTER(Sales, Sales[Year] = 2020)).

INTERSECT и EXCEPT принимают ровно две таблицы: INTERSECT(table1, table2), EXCEPT(table1, table2). Таблицы должны иметь одинаковую структуру. Порядок важен для EXCEPT — результат зависит от того, какая таблица первая. INTERSECT симметричен — порядок не влияет на результат.

NATURALINNERJOIN и NATURALLEFTOUTERJOIN: NATURALINNERJOIN(leftTable, rightTable). Столбцы с одинаковыми именами используются для соединения автоматически. Остальные столбцы добавляются к результату. При конфликте имен (одинаковые имена не участвующие в соединении) возникает ошибка.

Сравнение с альтернативами

В Excel для создания всех комбинаций нужны вспомогательные столбцы и формулы массива. Процесс трудоемкий и негибкий при изменении исходных данных. DAX CROSSJOIN создает комбинации динамически одной функцией. Изменение исходных данных автоматически обновляет результат.

SQL JOIN требует явного указания условий соединения через ON или USING. DAX естественные соединения автоматически определяют условия по именам столбцов. SQL более гибкий и явный, DAX — автоматизированный для типовых сценариев. Для сложных условий соединения в DAX используется FILTER над CROSSJOIN.

Pandas merge() и concat() в Python предлагают похожую функциональность с большим контролем над деталями. Но pandas работает с данными в памяти Python, DAX — с оптимизированной колоночной моделью. DAX эффективнее для аналитических запросов над большими данными.

Частые ошибки

Использование CROSSJOIN без предварительной фильтрации создает огромные таблицы, замедляющие вычисления. Всегда фильтруйте входные таблицы до минимально необходимого размера. Вместо CROSSJOIN(Products, Customers) используйте CROSSJOIN(VALUES(Products[ProductID]), VALUES(Customers[CustomerID])).

Несоответствие структур таблиц в UNION, INTERSECT, EXCEPT вызывает ошибки. Проверяйте количество и типы столбцов перед операцией. При необходимости используйте SELECTCOLUMNS для приведения таблиц к одинаковой структуре. Помните: порядок столбцов важен.

Неожиданные результаты естественных соединений из-за случайных совпадений имен столбцов. Таблицы могут иметь столбец Status, не предназначенный для соединения, но NATURALINNERJOIN использует его. Всегда проверяйте структуру таблиц и при необходимости переименовывайте столбцы.

Забывание о том, что виртуальные таблицы пересоздаются при каждом вычислении. Сложные виртуальные таблицы в часто используемых мерах могут существенно замедлить отчеты. Рассмотрите материализацию через вычисляемые таблицы для часто используемых структур.

## DAX-решение

```dax
// Все возможные комбинации категория-страна
[Theoretical Combinations] :=
COUNTROWS(
CROSSJOIN(
VALUES('DimProductCategory'[EnglishProductCategoryName]),  // уникальные категории
VALUES('DimGeography'[EnglishCountryRegionName])  // уникальные страны
)
)
// Фактические комбинации в продажах
[Actual Combinations] :=
COUNTROWS(
SUMMARIZE(
FactInternetSales,
'DimProductCategory'[EnglishProductCategoryName],
'DimGeography'[EnglishCountryRegionName]
)
)
// Процент покрытия рынка
[Market Coverage %] :=
DIVIDE([Actual Combinations], [Theoretical Combinations], 0)
```

Проверка результата: Теоретические комбинации: 24 (4 категории × 6 стран) Фактические комбинации: 24 (100% покрытие) По категориям: Bikes - 6 стран, Accessories - 6 стран Все категории представлены во всех странах модели При добавлении фильтров покрытие меняется динамически

## DAX-решение

```dax
// Клиенты обеих категорий (пересечение)
[Cross Category Customers] :=
COUNTROWS(
INTERSECT(
CALCULATETABLE(VALUES(FactInternetSales[CustomerKey]), 'DimProductCategory'[EnglishProductCategoryName] = "Bikes"),
CALCULATETABLE(VALUES(FactInternetSales[CustomerKey]), 'DimProductCategory'[EnglishProductCategoryName] = "Accessories")
)
)
// Эксклюзивные клиенты Bikes (разность)
[Bikes Only Customers] :=
COUNTROWS(
EXCEPT(
CALCULATETABLE(VALUES(FactInternetSales[CustomerKey]), 'DimProductCategory'[EnglishProductCategoryName] = "Bikes"),
CALCULATETABLE(VALUES(FactInternetSales[CustomerKey]), 'DimProductCategory'[EnglishProductCategoryName] = "Accessories")
)
)
```

Ответы

INTERSECT возвращает общие строки двух таблиц, EXCEPT — строки первой таблицы, отсутствующие во второй.

Естественные соединения автоматически определяют столбцы для связи по совпадению имен.

Да, CROSSJOIN может принимать несколько таблиц, создавая произведение всех.
